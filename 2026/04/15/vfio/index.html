VFIO：userspace 设备直通与虚拟机访问

发表于2026-04-15|更新于2026-04-16|内核与虚拟化
|浏览量:

本文是对 第十期 王佳琳的分享-VFIO 视频内容的整理和总结。

VFIO（Virtual Function I/O）是 Linux 内核提供的一个 IOMMU/设备无关框架，它允许将物理设备安全地暴露给 userspace，支持用户态驱动和虚拟机设备直通。

1. 为什么要用 VFIO

现代平台提供 DMA 和中断重映射能力，使 I/O 设备在宿主机中受限于分配给它们的边界。VFIO 利用这些硬件能力，为 userspace 提供受 IOMMU 保护的设备访问。

  • 用例
    • 用户态驱动，例如 DPDK 的 PMD
    • 虚拟机设备直通（device assignment）
  • 优势
    • 低延迟、高带宽
    • 可以直接使用裸机驱动
    • 支持复杂加速器、非 TCP/IP 网络适配器等场景

2. VFIO 的核心需求

要把一个物理设备呈现给虚拟机，必须满足两个方向的访问：

  1. 虚拟机访问设备

    • 访问 PCI 配置空间（必须模拟）
    • 访问 BAR 空间（可以通过 EPT/第二级页表实现直通）

  2. 设备访问虚拟机

    • 设备 DMA 访问虚拟机内存（可通过 DMA Remapping 直通）
    • 设备发送中断给虚拟机（可通过 Interrupt Posting 直通）

3. 为什么 PCI 配置空间不能直接直通

无论是 PIO 还是 MMIO，虚拟机访问物理设备的 PCI 配置空间都必须模拟。因为如果虚拟机直接改写真实设备的 PCI BAR 地址，虚拟机使用的是 GPA，而 GPA->HPA 的映射还没有建立，虚拟机无法访问 BAR 空间。

因此：

  • PCI 配置空间：由 VMM/QEMU 拦截模拟
  • BAR 空间：可以交给硬件直通访问

4. BAR 空间直通与地址映射

BAR 访问依赖于两级地址映射：

  • GVA -> GPA：由 guest kernel 建立
  • GPA -> HPA：由 host KVM/硬件建立

5. DMA remapping：设备访问虚拟机内存

设备需要通过 DMA 将数据直接读写到虚拟机内存。但虚拟机配置的 DMA 目标地址是 GPA，设备只能访问 HPA。DMA remapping 的作用就是在设备发起 DMA 请求时，将 GPA 转换成对应的 HPA，并且在转换过程中进行权限检查，确保设备只能访问被允许的内存区域。其是通过 IOMMU 硬件来实现的。

当设备要读写虚拟机内存时，设备侧发起的是 DMA 地址请求（可理解为 guest 视角的地址）。IOMMU 会查询该设备所属域（domain）的二级页表，把 GPA 转成 HPA；如果地址不在允许范围内，请求会被拒绝或报 fault，而不是直接打到任意物理内存。

和软件模拟路径相比，DMA remapping 的关键收益是数据面不再必须绕行 QEMU，减少一次用户态转发链路。

地址关系可以简化为：GPA（guest 物理地址） -> HPA（host 物理地址）。

设备做 DMA 时并不理解“这是哪台虚拟机的内存”，它只会发起总线地址访问请求。如果没有额外机制，设备可能直接写到错误的物理内存。因此会通过 DMA-remapping hardware 来实现地址转换和权限检查：当映射存在且权限允许时，DMA 请求会被放行并访问目标 HPA；当映射缺失或越权时，DMA 请求会被阻断并触发 fault，不会落到任意物理内存。

flowchart LR
  A[Device DMA Request] --> B[IOMMU]
  B --> C[Lookup domain page table]
  C --> D{Permission check}
  D -->|Pass| E[Translate GPA to HPA]
  E --> F[Access host memory]
  D -->|Fail| G[DMA fault and blocked]

隔离粒度由 IOMMU group 决定。group 是最小可隔离单元，同一 group 内设备无法彼此隔离，因此不能安全地分配给不同虚拟机；多个 group 可以放入同一 container 共享页表，以节省内存并降低 IOTLB miss，但这些 group 需要处于同一信任边界。

6. 中断转发：Interrupt Posting

设备完成 DMA 后需要尽快通知虚拟机。传统方式是“设备中断先到宿主机，再由 VMM 注入到虚拟机”，路径长、抖动大。

Interrupt Posting 的做法是让硬件按 vCPU 当前状态选择“直接投递”或“暂存后投递”。

当 vCPU 正在运行时，中断可直接投递给目标 vCPU；当 vCPU 被抢占时，中断先暂存，待该 vCPU 再次被调度后投递；当 vCPU 处于 hlt/idle 状态时，硬件会触发唤醒和调度，再完成中断投递。

flowchart TD
  A[Device raises interrupt] --> B{vCPU state}
  B -->|Running| C[Post directly to vCPU]
  B -->|Preempted| D[Store posted interrupt]
  D --> E[vCPU scheduled again]
  E --> F[Deliver interrupt]
  B -->|HLT or Idle| G[Wakeup and schedule via VMM]
  G --> F

7. VFIO vs 软件模拟 virtio

功能 软件模拟 virtio VFIO 直通
PCI 配置空间 QEMU 拦截模拟 QEMU 拦截模拟
BAR 空间 QEMU 拦截模拟 直通（部分场景仍可模拟）
设备 DMA 设备 DMA 访问宿主机内存，QEMU 再以共享内存方式访问 VM 直通（DMA Remapping）
中断注入 先由宿主机处理，再注入 VM 直通（Interrupt Posting）

VFIO 的缺点

  1. 虚拟机必须使用物理设备对应的真实驱动。不同宿主机上设备型号不同，会影响热迁移可行性。
  2. 设备 BAR 空间直通后扩大了攻击面，可能带来安全风险。
