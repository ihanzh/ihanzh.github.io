KV Cache（键值缓存）

发表于2026-05-12|更新于2026-05-12|LLM inference
|浏览量:

KV Cache 的作用

没有 KV Cache 时，朴素做法如下：

Step 1:  处理 tokens [0]              → 生成 token 1         （1 个 token）
Step 2:  处理 tokens [0, 1]           → 生成 token 2         （2 个 token）
Step 3:  处理 tokens [0, 1, 2]        → 生成 token 3         （3 个 token）
...
Step n:  处理 tokens [0, 1, ..., n-1] → 生成 token n         （n 个 token）

这有个明显的问题：每一步都要重新处理之前的所有 token，例如在 step2 时，模型需要计算 token 0 和 token 1 的 QKV 矩阵，而 step1 时已经计算过 token 0 的 QKV 矩阵了，这就造成重复计算，效率非常低。因此希望KV Cache 能够缓存之前计算过的 QKV 矩阵，在后续的步骤中直接复用，避免重复计算。

有了 KV Cache 后，如下图所示，每次只需要处理上一个 token，之前的 kv 矩阵都可以直接从缓存中复用，大大提升了效率：

并且我们计算 $q \cdot k^T \codt v$ 时，如果我们使用 kv cache 后，每次只需要取出最后一个 token 的 q 向量（红色部分），其结果就是不使用 kv cache 的最后一行，但同时模型输出下一个词的概率时，只取最后一行的结果即可，其他行是之前计算结果，负责预测之前的 token 的下一个词的概率，而新的 token 的下一个词的概率只需要最后一行的结果。这也就是我们为什么只需要缓存 K 和 V 矩阵，而不需要缓存 Q 矩阵的原因了。

next_logits = logits[:,-1,:]
next_token = sampler.sample(next_logits)

所以每一步计算，首先计算当前 token 的 kv 矩阵，然后取出之前缓存的 kv 矩阵，拼接在一起，得到新的 kv 矩阵，最后计算 q·k^T·v 的结果，最后还需要把新的 kv 矩阵缓存起来，供下一步使用。

并且使用 kv cache 后，可以发现 $q \cdot k^T$ 的结果只有一行，并且这一行的所有数据都需要，因此 decode 阶段就不需要 causal mask 了，同理每一步也只需要计算当前 position 的旋转位置编码了，不需要计算之前 token 的旋转位置编码。

DynamicKVCache 实现

最简单的思路就是使用一个列表来缓存每一层的 kv 矩阵，每一步计算时更新对应层的 kv 矩阵：

class DynamicKVCache:

    def __init__(self, num_layers: int):
        self.key_cache: list[torch.Tensor|None] = [None] * num_layers
        self.value_cache: list[torch.Tensor|None] = [None] * num_layers
        self._seq_len = 0

    def get_seq_len(self) -> int:
        return self._seq_len

    def update(
        self,
        layer_idx: int,
        # 当前 step 计算得到的 kv 矩阵，形状为 (batch, num_heads, 1, head_dim)
        key_states: torch.Tensor,
        value_states: torch.Tensor,
    ) -> tuple[torch.TEnsor, torch.Tensor]:
        # 从缓存中取出之前的 kv 矩阵
        cached_keys = self.key_cache[layer_idx]
        cached_values = self.value_cache[layer_idx]

        if cached_keys is None:
            full_keys = key_states
            full_values = value_states
        else:
            # 将之前的 kv 矩阵和当前 step 计算得到的 kv 矩阵拼接在一起，得到新的 kv 矩阵
            full_keys = torch.cat([cached_keys, key_states], dim=2)
            full_values = torch.cat([cached_values, value_states], dim=2)

        # 将新的 kv 矩阵缓存起来，供下一步使用
        self.key_cache[layer_idx] = full_keys
        self.value_cache[layer_idx] = full_values


        if layer_idx == 0:
            self._seq_len = full_keys.shape[2]

        return full_keys, full_values

    def clear(self) -> None:
        for layer_idx in range(len(self.key_cache)):
            self.key_cache[layer_idx] = None
            self.value_cache[layer_idx] = None
        self._seq_len = 0

这个思路实现了 KV Cache 的基本功能，但每一步都要重新申请内存完成拷贝和拼接，效率较低。更高效的实现方式是预先申请一个足够大的内存空间，每一步计算时直接将新的 kv 矩阵写入到对应位置，这样就避免了每一步都要重新申请内存和拷贝数据了，但这个方法需要事先知道最大序列长度，或者在达到最大序列长度时进行扩容。目前业界的主流实现是使用分页的 kv cache，类似于操作系统中的分页机制，预先申请一个较大的内存空间，分成多个页，每页可以存储一定数量的 kv 矩阵，当需要使用新的页时再进行申请，这样就可以在不事先知道最大序列长度的情况下实现高效的 kv cache 了。

分页 KV Cache 实现

核心组件：

  • MHAKVCache：底层 KV 物理池（只做物理存储），直接操作物理页。
  • block_table：逻辑 token 下标到物理 page 下标的映射。
  • CacheManager：负责管理物理页的申请和回收。

三ze个组件的关系如下：

每有一个 token，block_table 通过 CacheManager 申请一个物理页，然后计算该 token 的 kv 矩阵并写入到 MHAKVCache 中对应的物理页中，后续需要使用 kv 矩阵时，通过 block_table 找到对应的物理页下标，从 MHAKVCache 中取出 kv 矩阵供计算使用。

MHAKVCache 实现

LLM 初始化时，生成一个 MHAKVCache 实例，预先申请一个较大的内存空间，分成多个页，每页可以存储一定数量的 kv 矩阵：

self.num_pages = 2048
self.mha_kv_cache = MHAKVCache(
    num_kv_heads=config.num_kv_heads,
    num_layers=config.num_layers,
    head_dim=config.head_dim,
    num_pages=self.num_pages,
    dtype=self.dtype,
    kv_layout=KVCacheLayout.LayerFirst,
    device=self.device,
    page_size=1,
)

MHAKVCache 只负责物理存储，其申请的物理内存空间的形状根据 kv_layout 的不同而不同：

大小：2 是 key 和 value 两种类型，num_pages 是页数，num_layers 是层数，local_kv_heads 是每层的 kv 头数，page_size 是每页存储的 token 数量，head_dim 是每个头的维度。

PageFirst: 按页划分，并做一个 permute(只改维度顺序，不会像 reshape 那样按新顺序复制数据，通常是 view) 操作，将第 2 和第 3 个维度交换，方便后续按页访问

LayerFirst: 按 layer 划分

match kv_layout:
    case KVCacheLayout.PageFirst:
        kv_buffer = torch.empty(
            (2, num_pages, num_layers, local_kv_heads, page_size, head_dim),device=device,
            dtype=dtype,
        ).permute(0, 2, 1, 3, 4, 5)
    case KVCacheLayout.LayerFirst:
        kv_buffer = torch.empty(
            (2, num_layers, num_pages, local_kv_heads, page_size, head_dim),
            device=device,

        )
    case _:

然后提供取缓存和存缓存的接口：

def k_cache(self, index: int) -> torch.Tensor:
    return self._k_buffer[index]

def v_cache(self, index: int) -> torch.Tensor:
    return self._v_buffer[index]

def store_kv(
    self,
    layer_id: int,
    out_loc: torch.Tensor,
    k: torch.Tensor,
    v: torch.Tensor,
) -> None:

    if self.page_size == 1:
        self._k_buffer[layer_id, out_loc, :, 0, :] = k
        self._v_buffer[layer_id, out_loc, :, 0, :] = v

    else:
        page_idx = out_loc // self.page_size
        slot_idx = out_loc % self.page_size

        self._k_buffer[layer_id, page_idx, :, slot_idx, :] = k
        self._v_buffer[layer_id, page_idx, :, slot_idx, :] = v

        kv_cache = None
        paged_block_table = None
        paged_seq_len = 0

        if use_paged_kv_cache:
            paged_block_table = self.cache_manager.allocate(input_ids.shape[1])
        elif use_kv_cache:
            kv_cache = DynamicKVCache(self._num_layers)

block_table 实现

block_table 是一个列表，长度为当前生成的 token 数量，每个元素是一个元组，包含物理 page 下标和页内位置下标：

开始 prefill 时，paged_block_table 就是申请 token 长度个页

if use_paged_kv_cache:
    paged_block_table = self.cache_manager.allocate(input_ids.shape[1])

然后 decode 阶段每生成一个 token，block_table 就申请一个新的页并拼接到 block_table 中：

if use_paged_kv_cache:
    new_block = self.cache_manager.allocate(1)
    paged_block_table = torch.cat([paged_block_table, new_block])

然后计算 kv cache 时：

  1. 从 block_table 中取出当前 token 对应的物理 page 下标
  2. 将当前 token 的 kv 矩阵写入到对应的物理 page 中
  3. 从 block_table 中取出当前 token 之前的所有 token 对应的物理 page 下标
  4. 从物理页中取出对应的 kv 矩阵，得到新的 kv 矩阵，供当前 token 的 attention 计算使用：
if paged_kv_cache is not None:
    # for this token, the k/v will be written to the location pointed by paged_block_table[paged_seq_len]
    out_loc = paged_block_table[paged_seq_len: paged_seq_len + S]

    # k/v: (B, num_kv_heads, 1, head_dim)
    # k[0]: (num_kv_heads, 1, head_dim)
    # k_to_store: (num_kv_heads, head_dim)
    k_to_store = k[0].transpose(0, 1)
    v_to_store = v[0].transpose(0, 1)

    paged_kv_cache.store_kv(self._layer_idx, out_loc, k_to_store, v_to_store)

    # 取出所有的kv，供当前token的attention计算使用
    all_locs = paged_block_table[:paged_seq_len + S]

    v = paged_kv_cache.v_cache(self._layer_idx)[all_locs, :, 0, :].transpose(0, 1).unsqueeze(0)

CacheManager 实现

负责管理哪些物理页是空闲的，哪些物理页是被占用的，提供申请和回收物理页的接口，主要靠 free_slots 来管理空闲页，初始时 free_slots 包含所有页的下标，当申请页时，从 free_slots 中取出需要的页的下标，当回收页时，将回收的页的下标添加到 free_slots 中：

def _free(self, indices: torch.Tensor) -> None:
    if len(indices) > 0:
        self._free_slots = torch.cat([self._free_slots, indices])

def allocate(self, need_len: int) -> torch.Tensor:

    if need_len <= (free_len := len(self._free_slots)):
        allocted = self._free_slots[:need_len]
        self._free_slots = self._free_slots[need_len:]
        return allocted
    raise NotImplementedError("CacheManager eviction is not implemented.")
