Containerd 与 Runtime shim

[ 编排调度层 ]                  Kubernetes
                                   │ (CRI 协议)
[ 高级管理层 ]          containerd  /  CRI-O  /  Docker  /  Podman
                                   │ (OCI 协议)
             ┌─────────────────────┼─────────────────────┐
[ 运行时层 ] │ 标准共享型          │ 微虚机隔离型        │ 用户态沙箱型
             │ runc / crun         │ Kata Containers     │ gVisor
             └──────┬──────────────┴──────┬──────────────┴──────┬──────┘
[ 隔离引擎层 ]      │                     │                     │ (拦截)
             Namespaces & Cgroups  Firecracker / QEMU    Sentry (Go态内核)
                    │                     │                     │
[ 物理内核层 ]      ▼                     ▼                     ▼
             Host Linux Kernel          KVM               Host Linux Kernel

containerd 对上层暴露统一 CRI 语义，通过 OCI Runtime 接口调用具体 runtime（runc/kata/gVisor）。Kata shim 负责将 containerd 的 sandbox 和 task 请求转为内部运行时调用，提供更高隔离能力。

shim 启动路径：从 start 到 run

在 shim 的 main.rs 中，parse_args() 解析命令行参数，得到 Action::Start/Run/Delete 等不同的操作类型。

crates/shim/src/bin/main.rs
match action {
	Action::Start(args) => ShimExecutor::new(args).start().context("shim start")?,
	Action::Delete(args) => {
		...
	}
	Action::Run(args) => {
		...
	}
	Action::Help => show_help(&args[0]),
	Action::Version => show_version(None),
	Action::Info => show_info().context("show info")?,
}

do_start 路径中，shim 会根据 ContainerType 来决定如何创建 socket 和进程：

  • PodSandbox: 创建独立 sandbox
  • PodContainer: 挂载到已有 sandbox
  • SingleContainer: 直连容器模式

主要看 PodSandbox 和 SingleContainer 这两种情况，都会创建一个新的 socket，然后 fork exec 一个新的进程，并将 socket fd 传递给子进程。

crates/shim/src/shim_start.rs
fn do_start(&mut self) -> Result<PathBuf> {
	...
	match container_type {
		ContainerType::PodSandbox | ContainerType::SingleContainer => {
			let address = self.socket_address(&self.args.id)?;
			let socket = new_listener(&address)?;
			let child_pid = self.create_shim_process(socket)?;
			self.write_pid_file(&bundle_path, child_pid)?;
			self.write_address(&bundle_path, &address)?;
			Ok(address)
		}
		ContainerType::PodContainer => {
			...
		}
	}
}

在 create_shim_process() 中，shim 会设置环境变量 KATA_RUNTIME_BIND_FD 为 socket fd，然后 fork exec 自己的二进制来运行 run 操作，并且没有传递 rest_args，而在前文的 parse_args() 中，如果没有 rest_args 就默认是 Action::Run，因此子进程会直接进入 run 分支。

run 进程监听并提供 ttrpc 服务

在 run 分支中，shim 会创建 ENV_TOKIO_RUNTIME_WORKER_THREADS 个线程的 Tokio runtime，然后调用 ServiceManager::new() 来创建一个新的 ServiceManager 实例，并且传入 socket fd 来创建 ttrpc server。最后调用 service_manager.run().await 来启动事件循环，等待 containerd 的 RPC 请求。

crates/shim/src/shim_run.rs/ServiceManager::new()
let (sender, receiver) = channel::<Message>(MESSAGE_BUFFER_SIZE);
let rt_mgr = RuntimeHandlerManager::new(id, sender).context("new runtime handler")?;
let handler = Arc::new(rt_mgr);
let mut server = unsafe { Server::from_raw_fd(task_server_fd) };
server = server.set_domain_unix();

在 ServiceManager::run() 中，shim 会通过 registry_service() 来注册 sandbox 和 task 的 ttrpc 服务，然后调用 start_service().await 来启动 ttrpc server。之后进入一个事件循环，等待 receiver 中的消息。

crates/shim/src/shim_run.rs/ServiceManager::run()
self.registry_service().context("registry service")?;
self.start_service().await.context("start service")?;

info!(sl!(), "wait server message");
let mut rx = self.receiver.take();
if let Some(rx) = rx.as_mut() {
	while let Some(r) = rx.recv().await {
		...
	}
}

在 registry_service 中，将 sandbox 和 task 的服务注册到 ttrpc server 中

crates/service/src/manager.rs/registry_service()
let sandbox_service: Arc<dyn sandbox_async::Sandbox + Send + Sync> =
    Arc::new(SandboxService::new(self.handler.clone()));
let s = s.register_service(sandbox_async::create_sandbox(sandbox_service));

let task_service: Arc<dyn shim_async::Task + Send + Sync> =
	Arc::new(TaskService::new(self.handler.clone()));
let s = s.register_service(shim_async::create_task(task_service));
self.server = Some(s);

ttrpc 请求分发流程

registry_service() 注册服务：

  • sandbox_async::create_sandbox(Arc::new(SandboxService::new(handler)))
  • shim_async::create_task(Arc::new(TaskService::new(handler)))

create_sandbox/create_task 生成 Service，按方法名映射到 MethodHandler

SandboxService：处理“沙箱级”生命周期（VM/沙箱层面），对应容器组级别的隔离环境（guest VM、sandbox 进程等）
TaskService：处理“容器级”任务（具体 container/task），对应 sandbox 里具体进程/容器操作

请求到达时：

  1. ttrpc 根据 service（如 containerd.runtime.sandbox.v1.Sandbox）定位 Service
  2. 找到 method（如 CreateSandbox）对应 MethodHandler
  3. 执行 MethodHandler::handler(ctx, req)
  4. async_request_handler! 宏处理：
    • 反序列化请求消息
    • 调用 self.service.<method>(ctx, req)（业务实现）
    • 序列化响应返回

crates/service/src/sandbox_service.rs 和 task_service.rs：

例如 sandbox_service：

  • impl_service! 宏将 sandbox_async::Sandbox trait 方法映射到 handler_message
  • SandboxService::handler_message 调用到 RuntimeHandlerManager 的 handler_sandbox_message：

RuntimeHandlerManager::handler_sandbox_message：

crates/runtimes/src/manager.rs/RuntimeHandlerManager::handler_sandbox_message()
#[instrument(parent = &*(ROOTSPAN))]
pub async fn handler_sandbox_message(&self, req: SandboxRequest) -> Result<SandboxResponse> {
	if let SandboxRequest::CreateSandbox(sandbox_config) = req {
		let config = sandbox_config.deref().clone();

		self.sandbox_init_runtime_instance(config)
			.await
			.context("init sandboxed runtime")?;

		Ok(SandboxResponse::CreateSandbox)
	} else {
		self.handler_sandbox_request(req)
			.await
			.context("handler request")
	}
}

这些处理会向 shim 主线程 send msg，主线程通过 receiver 触发 service 的处理逻辑：

crates/service/src/manager.rs/ServiceManager::run()
Action::Start => self.start_service().await.context("start listen"),
Action::Stop => self.stop_service().await.context("stop listen"),
Action::Shutdown => {
	self.stop_service().await.context("stop listen")?;
	break;
}
Action::Event(event) => {
	info!(sl!(), "get event {:?}", &event);
	self.event_publisher
		.forward(event)
		.await
		.context("forward event")
}
  • 触发 start_service()（当前 ttrpc 会重新 start 监听，比如容器 sandbox 重新上线）
  • 触发 stop_service()（停止监听 RPC，用于暂停/下线）
  • 先 stop_service()，然后 break 跳出循环（shim 彻底关闭事件循环）
  • 不是控制 service 监听，而是业务事件转发给 event_publisher.forward(event)（例如 TaskCreate/TaskStart/TaskDelete 通知上层系统）
