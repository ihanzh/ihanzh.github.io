DMA 原理和工作流程

发表于2026-03-16|更新于2026-03-16|device
|浏览量:

在计算机系统中，数据搬运是一项极其频繁且繁重的任务。如果每一次硬盘读取、网卡收发都要 CPU 亲自去一字节一字节地搬运，哪怕 CPU 算力再强，也会被这些“体力活”彻底拖垮。

为了解放 CPU，DMA（Direct Memory Access，直接内存访问） 技术应运而生。

一、 核心概念：什么是 DMA 与 DMA Buffer？

1. DMA（直接内存访问）

本质上，DMA 是一种硬件级的数据旁路机制。它的核心是一个名叫 DMAC（DMA控制器） 的硬件（现在通常集成在主板芯片组或外设内部，如 PCIe 设备的 Bus Master）。

简单来说，它的工作就是：“把数据搬运这种繁重的体力活从 CPU 手里接管过来，让 CPU 去干更有价值的计算工作。”
在使用 DMA 的场景下，CPU 只需要在开头交代一下任务（源地址、目的地址、传输大小），在结尾接收一下中断（验收成果），中间的数据流转完全由 DMA 硬件独立完成。

2. DMA Buffer（DMA 缓冲区）

从内存（Memory）的视角来看，DMA Buffer 是操作系统在系统主存（RAM）中特意开辟出来的一块特殊区域。可以理解为 CPU 和 外设（Device）之间交接货物的专属 VIP 卸货平台。

当设备需要写数据（比如网卡收到网络包），设备的 DMA 引擎就会把数据从设备内部拷贝到内存的这块 DMA Buffer 中；反之，当设备要读数据（比如网卡发包），它就会从 DMA Buffer 中把数据吸走。

二、 特别注意点（深入底层的 3 个大坑）

理解 DMA，绝不能只停留在“搬运工”的比喻上，必须跨过以下三个底层门槛：

⚠️ 1. Cache 一致性（Cache Coherence）问题 —— 最容易翻车的地方

现代 CPU 和主存之间隔着 L1/L2/L3 缓存，CPU 读写数据通常是在操作 Cache。这就导致了 CPU 看到的数据和 DMA（直接连着主存）看到的数据可能不一致！

  • CPU 发数据给外设时：CPU 刚把数据写到内存地址 X，但实际上数据可能还暂存在 CPU Cache 里。此时如果 DMA 去主存取数据，取到的将是老旧的错误数据。
    • 解决：启动 DMA 前，CPU 必须执行 Flush（刷出），强制把 Cache 写回主存。
  • 外设发数据给 CPU 时：外设通过 DMA 偷偷把新数据写到了主存的地址 Y。但 CPU 的 Cache 里如果还有地址 Y 的历史遗留数据，CPU 就会直接读 Cache 而无视主存的新数据。
    • 解决：CPU 读取前，必须执行 Invalidate（失效），强制废弃当前 Cache，去主存重新拉取。

⚠️ 2. 物理地址与虚拟地址的鸿沟

传统的 DMA 硬件非常“死板”，它只认物理地址，根本不知道什么是虚拟内存、什么是页表。如果内核随便给它一个虚拟地址，它可能会直接把数据写坏别人的物理内存空间。

  • 解决 1（连续物理内存）：早期操作系统分配 DMA Buffer 时，必须保证这块内存在物理上是绝对连续的（例如 Linux 的 CMA 机制）。
  • 解决 2（Scatter-Gather DMA）：外设支持读取包含多个非连续物理页地址的“描述符链表”，将数据打散存储。
  • 解决 3（IOMMU）：现代主板引入了 IOMMU 硬件，相当于给 DMA 专门配了一个“虚拟地址翻译官”，设备可以直接使用 IO 虚拟地址（IOVA）进行安全传输。

⚠️ 3. 内存必须被“锁定（Pinned）”

普通程序的物理内存如果不用，可能被操作系统偷偷换出（Swap）到硬盘上。但 DMA Buffer 绝对不行！
一旦 DMA Buffer 被换出，原来的物理地址分配给了其他进程，而外设毫不知情还在往那个物理地址写数据，就会导致灾难性的内存覆盖（甚至内核崩溃）。因此，DMA Buffer 所在的物理页必须被死死锁定在内存中。

三、 工作流程深度拆解（以网卡接收数据为例）

为了让过程更具象，我们以 “网卡接收网络数据包（设备写数据到内存）” 为例，看看一整套 DMA 流程到底发生了什么：

Step 1: 预留车位（内核初始化）

在网卡还没收到任何数据之前，网卡驱动程序会在内核空间里申请一块物理地址连续的 DMA Buffer，并查到它的物理首地址（假设为 0x1000）。驱动将 0x1000 写入网卡的寄存器，告诉它：“数据来了就送到这里”。

Step 2: 货物到达（设备内部暂存）

一根网线传来了光电信号，网卡芯片解析出一个 1500 字节的包，暂时放在网卡内部极小的 SRAM 缓存（FIFO）中。

Step 3: 疯狂搬砖（DMA 传输数据）

网卡发现 FIFO 有数据，立即启动其内部的 DMA 引擎（Bus Master）。DMA 引擎绕过 CPU，直接通过 PCIe 总线向主板发起内存写请求，将这 1500 字节的数据一点点“挤”进物理内存地址 0x1000 开始的 DMA Buffer 里。

Step 4: 汇报工作（硬件中断）

数据全部写进 DMA Buffer 后，网卡通过系统总线向 CPU 发送一个硬件中断信号。相当于快递员把货放进了储物柜，然后按响了你家的门铃。

Step 5: CPU 验收（处理 Cache 并读取）

  • CPU 收到中断，暂停当前任务来处理网卡事件。
  • CPU 先执行 Cache Invalidate（使缓存失效），确保自己不去读旧的缓存。
  • CPU 从系统主存的 DMA Buffer 中读取这 1500 字节的真实网络包数据。
  • 操作系统内核剥离网络包的 TCP/IP 报头，最后通过 Socket 将数据交给用户层的应用程序（如浏览器、微信）。

四、DMA remapping 与 IOMMU 的引入

在早期的电脑里，设备的 DMA 引擎是直接使用**系统物理地址（Physical Address, PA）**来读写内存的。这带来了三个致命问题：

  1. 灾难一：安全漏洞（DMA 攻击 / 内存越界）
    既然外设可以直接读写物理内存，那如果网卡驱动写错了，或者插上了一个恶意的 PCIe 设备（比如黑客特制的黑客U盘/Thunderbolt设备），它就可以通过 DMA 悄悄读取操作系统的内核密码，或者直接把恶意代码写进操作系统的核心物理内存里！这叫 “DMA 攻击 (DMA Attack)”。系统对此毫无防备，因为 DMA 直接绕过了 CPU！
  2. 灾难二：32位老设备的“残疾”
    很多老旧的外设（比如老的 32 位声卡或网卡），它的 DMA 引擎最多只能寻址 $2^{32}$ = 4GB 的物理内存。如果你电脑装了 16GB 内存，而操作系统刚好把 DMA Buffer 分配在了 8GB 的位置，这个老设备就“够不着”了。为了解决这个问题，内核只能搞个低地址的“替身缓冲区（Bounce Buffer）”，让 CPU 疯狂倒腾数据，性能大打折扣。
  3. 灾难三：虚拟机（VM）无法独占物理硬件
    在云计算时代，你想把主机的物理显卡（GPU）或者物理网卡直接分配给里面的一个虚拟机（这叫 PCIe Passthrough / 直通技术）。但是，虚拟机操作系统看到的“物理地址”其实是假的（是宿主机给它的虚拟地址）。如果虚拟机把这个“假物理地址”交给真实的物理显卡去搞 DMA，显卡一写，主机的真实内存就全乱套了！

什么是 DMA Remapping？

为了解决这些问题，Intel（VT-d 技术）和 AMD（AMD-Vi 技术）在主板的芯片组里塞进了一个新硬件：IOMMU。

DMA Remapping 的核心思想就是：给所有外设也配上一个“虚拟地址翻译官”和“安检员”。

有了 DMA Remapping 之后，外设看到的不再是真实的物理内存地址，而是一个叫做 IOVA（I/O 虚拟地址，I/O Virtual Address） 的东西。

它的工作原理如下：

  1. 分配 IOVA： 操作系统给网卡分配了一块 DMA Buffer（实际物理地址是打散的 0x5000 和 0x9000）。但操作系统通过 IOMMU 给网卡发了一个连续的虚拟地址（IOVA）：0x1000 到 0x2000。
  2. 设备发起 DMA： 网卡的 DMA 引擎开始干活，它以为自己往连续的 0x1000 物理地址写数据。
  3. IOMMU 拦截与翻译（重映射）： 网卡的读写请求在主板 PCIe 总线上跑的时候，被 IOMMU 硬件一把拦住。IOMMU 查阅自己内部的“页表（Page Table）”，将设备的 IOVA 0x1000 **重映射（Remap）**成真实的物理地址 0x5000。
  4. 安全放行： IOMMU 还会检查：“这个网卡有权限写 0x5000 吗？”如果有，放行并写入真实内存；如果没有（比如恶意设备想写内核区域），IOMMU 会直接阻断这个 DMA 请求，并向 CPU 报告一个严重错误。

DMA Remapping 带来的收益

1. 安全隔离（IOMMU Protection）

每个设备都有自己独立的 IOVA 空间。网卡只能读写网卡专属的虚拟地址，显卡只能读写显卡的。任何设备想越界访问其他物理内存，都会被 IOMMU 拦截。彻底终结了 DMA 攻击！

2. 化零为整，消灭内存碎片

现代系统很难找到大块连续的物理内存。有了重映射，操作系统可以把物理上支离破碎的内存页，通过 IOMMU 映射成一段连续的 IOVA 给设备用。
对外设来说：爽！我拿到了一大块连续的地址！
对系统来说：爽！我不用费劲去整理内存碎片了！

3. 拯救 32 位老设备

老设备只能访问 4GB 以下的地址？没关系！操作系统给它分配一个 4GB 以下的 IOVA（比如 0x80000000），然后 IOMMU 在底层把这个地址重映射到物理内存的 12GB 处。老设备浑然不知，却完美用上了高位内存，告别了低效的 Bounce Buffer（替身缓冲区）。

4. 云计算的基石：硬件直通（Passthrough）

这是 DMA Remapping 最具商业价值的地方。
当宿主机把物理网卡直通给虚拟机时，虚拟机操作系统会把它以为的“物理地址 (GPA)” 给网卡。此时，宿主机的 IOMMU 会自动把这个 GPA 重映射为宿主机的真实物理地址 (HPA)。
物理硬件就可以直接和虚拟机高速交换数据，性能几乎 0 损耗！

五、DMA remapping 的隔离机制

在一台物理服务器上，无论跑了多少个虚拟机（VM），物理主板上的 IOMMU 硬件通常只有一个（或少量的几个节点）。

既然所有的网卡、显卡直通（Passthrough）给不同的虚拟机时，全都要经过宿主机（Host）的这个唯一的 IOMMU 进行重映射，那么，这个单一的 IOMMU 凭什么能把 VM_A 的设备和 VM_B 的设备完美隔离开来，保证它们不会互相读写内存呢？

这就引出了 DMA 重映射实现隔离的三大底层核心机制：

机制一：身份识别（Source ID / BDF）—— 查“发件人”

IOMMU 要做隔离，第一步必须知道：“现在的这个 DMA 请求，到底是哪个外设发出来的？”

在 PCIe 总线架构中，每一个硬件设备都有一个全球（全系统）唯一的编号，叫做 BDF（Bus:Device.Function，总线号:设备号.功能号）。
你可以把它理解为每个设备的**“身份证号”或“发件人号码”**。

当物理网卡（直通给 VM_A）向主板发起一次 DMA 内存读写请求时，这个请求的硬件数据包（PCIe TLP 报文）里，必然会强行带上它自己的 BDF 号（Source ID）。
IOMMU 硬件就守在总线路口，每次拦下 DMA 请求，第一件事就是提取这个 BDF 号。

机制二：上下文表（Context Table）与保护域（Domain）—— 查“专属字典”

既然知道了是哪个设备发出的，IOMMU 怎么决定用哪套规则呢？

宿主机（Hypervisor / Host OS，比如 KVM/ESXi）在启动虚拟机、分配设备时，会在物理内存里帮 IOMMU 建一套极其严密的多级查询表（Root Table -> Context Table）。

  1. 建立保护域（Protection Domain）： 宿主机给 VM_A 分配一个独立的域（Domain A），给 VM_B 分配一个独立的域（Domain B）。
  2. 分配专属页表（Page Table）： 宿主机为 Domain A 创建一套专属的 “GPA 到 HPA（客户机物理地址 -> 宿主机物理地址）” 的翻译页表；同理为 Domain B 也创建一套。
  3. 绑定 BDF 与页表： 宿主机把设备的 BDF 填进 IOMMU 的上下文表里。
    • 告诉 IOMMU：“只要看到 BDF 是 03:00.0（网卡），它属于 Domain A，请查 A 的页表！”
    • “看到 BDF 是 04:00.0（显卡），它属于 Domain B，请查 B 的页表！”

这就是隔离的核心原理：
因为 VM_A 的设备只能使用 Domain A 的页表，而 Domain A 的页表里，绝对不可能包含 VM_B 的物理内存地址（HPA）。
如果 VM_A 的黑客控制网卡，试图生成一个指向 VM_B 内存的虚拟地址，IOMMU 在 Domain A 的页表里根本查不到这个映射记录（或者权限被标记为不可写），就会直接报 IOMMU Fault，把这个 DMA 攻击当场击毙！

机制三：IOMMU 组（IOMMU Group）与 ACS 技术 —— 堵住“抄近道”的漏洞

前面的逻辑听起来很完美，但现实中的硬件拓扑非常复杂，这里有一个极其危险的漏洞。

假设主板上有一个 PCIe 交换机（PCIe Switch），下面插着两个设备：网卡 1（给 VM_A）和网卡 2（给 VM_B）。
在 PCIe 规范里，有一个机制叫 Peer-to-Peer（端到端通信）。网卡 1 发出的 DMA 请求，如果在 PCIe 交换机这一层发现目标地址刚好是网卡 2，交换机会直接把数据抄近道发给网卡 2，根本不往上层送，彻底绕过了主板上的 IOMMU！
如果发生了这种事，隔离就成了笑话，VM_A 就能直接攻击 VM_B 的设备了。

为了堵住这个漏洞，实现真正的 DMA 隔离，硬件和 Linux 内核引入了两个关键概念：

  1. PCIe ACS（Access Control Services，访问控制服务）：
    这是一种高级 PCIe 硬件特性。宿主机会强行开启主板和 PCIe 交换机的 ACS 功能。它的作用是：禁止任何底层的“抄近道”行为。所有设备的 DMA 请求，无论发给谁，必须全部“滚上去”给 IOMMU 审查一遍。不支持 ACS 的主板，做虚拟机直通是非常危险的。
  2. IOMMU Group（Linux VFIO 机制）：
    这是宿主机内核里管理隔离的最小单位。
    如果系统发现几个设备共享同一个不支持 ACS 的 PCIe 桥（它们可能互相串通绕过 IOMMU），Linux 内核就会把这几个设备强制绑定进同一个 IOMMU Group。
    规则是极其霸道的： 一个 IOMMU Group 里的所有设备，要么全部直通给同一个虚拟机（VM_A），要么全留在宿主机。绝对不允许把同一个 Group 里的设备拆开分给不同的虚拟机。这就从物理拓扑上彻底杜绝了跨虚拟机的 DMA 泄露。

结语

DMA 的本质是一场 “权利的下放”。通过硬件层面的数据旁路机制、专属的 DMA 缓冲区以及复杂的 Cache 协调机制，计算机系统实现了外设与内存之间的高速直连。而 DMA Remapping 和 IOMMU 的引入，则是对这一机制的安全加固和功能扩展，让我们在享受高性能的同时，也能高枕无忧地使用各种外设，甚至在云计算时代实现虚拟机的安全直通。

本文是对 PCIe 原理和资源访问 视频内容的整理和总结。 一、 PCIe 基本原理与通信机制 PCIe 的设计与早期的 PCI 总线有着本质的区别。 拓扑结构差异：传统的 PCI 采用的是总线拓扑，设备之间需要进行总线仲裁，并且共享带宽 。而 PCIe 采用的是树形拓扑，实现了端到端的通信，每个设备都能独享链路带宽 。 物理层通信：PCIe 的两个端点通过双 Link 进行全双工传输 。其中一个 Link 负责发送（Tx），另一个负责接收（Rx），这两个 Link 共同组成了一个 Lane（通道） 。 灵活的带宽配置：一条 PCIe 传输链路可以由多个 Lane 组成，系统支持 1、2、4、8、16 甚至 32 个 Lane，以此来灵活匹配不同设备对带宽的需求 。 二、 PCIe 核心组件架构 在 PCIe 的树形拓扑中，有三个核心组件构成了整个网络 ： Root Complex (RC)：这是连接 CPU、内存与 PCIe 树的“根节点”，它不仅负责 CPU/内存与 PCIe 设备之间的数据传输，还处理设备间的 P2P 通信，并实现 IOMMU 功能 。...
