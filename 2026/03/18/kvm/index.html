KVM：通过匿名 inode 实现 VM fd 的文件操作

发表于2026-03-18|更新于2026-03-22|内核与虚拟化
|浏览量:

背景：/dev/kvm 与 ioctls

在 Linux 中，用户态通过打开 /dev/kvm 来访问 KVM 功能。内核中对应的设备驱动注册了一个 character device，其 file_operations 定义了 unlocked_ioctl，指向 kvm_dev_ioctl()：

static const struct file_operations kvm_chardev_ops = {
    .owner          = THIS_MODULE,
    .unlocked_ioctl = kvm_dev_ioctl,
    .llseek         = noop_llseek,
    KVM_COMPAT(kvm_dev_ioctl),
};

当用户态调用 ioctl(fd, KVM_CREATE_VM, type) 时，内核走到了 kvm_dev_ioctl()，并最终创建了一个新的 VM 对象。

1. KVM_CREATE_VM: 创建一个新的 VM fd

调用流程大致如下：

kvm_dev_ioctl()
|
|__ kvm_dev_ioctl_create_vm(type)
    fd = get_unused_fd_flags(O_CLOEXEC)
    kvm = kvm_create_vm()
    file = anon_inode_getfile("kvm-vm", &kvm_vm_fops, kvm, O_RDWR);
    fd_install(fd, file)

关键几点：

  • kvm_create_vm() 在内核里创建并初始化一个 struct kvm *（即一个 VM 对象）。

  • get_unused_fd_flags(O_CLOEXEC) 分配一个新的 fd（用户态会拿到这个 fd）。

  • anon_inode_getfile("kvm-vm", &kvm_vm_fops, kvm, O_RDWR) 创建一个 struct file，并将其绑定到一个“匿名 inode”。

    anon_inode_getfile() 并不创建一个真实的磁盘文件，它创建的是一个“匿名 inode 文件”：

    • 不是磁盘文件：不在文件系统中，没有路径名。
    • 共享 inode：多个匿名文件可以共享同一个 inode（节省资源）。
    • 允许内核向用户空间暴露一个 fd：用户可以对它做 ioctl、close 等常规文件操作。

    参数说明：

    • "kvm-vm"：在 /proc/<pid>/fd/<n> 中显示为该名称（即 fd 的 class 名称）。
    • &kvm_vm_fops：这个 fd 的行为由对应的 file operations 决定（包括 ioctl、release 等）。
    • kvm：作为 file->private_data 的私有数据，后续 ioctl/close 会用到它。
    • O_RDWR：fd 的访问模式。

    创建 VM fd 时绑定的 struct file 的 ->private_data 就是 struct kvm *。
    VM fd 的 file ops 是 kvm_vm_fops，其 unlocked_ioctl 指向 kvm_vm_ioctl。
    在 kvm_vm_ioctl() 的开头，通常会这样取出 VM：

    struct kvm *kvm = filp->private_data;

    之后的所有 KVM_* ioctl 都会基于这个 kvm 进行处理，因此同一个 fd 上的 ioctl 永远作用在同一个 VM 对象上。

  • 通过 fd_install(fd, file)，把这个内核 struct file 和用户态的 fd 关联起来。

从此，用户拿到的 fd 与内核中的那个 VM 对象（struct kvm）建立了联系。

2. KVM_CREATE_VCPU: 创建 vCPU fd

用户态通过 ioctl(vm_fd, KVM_CREATE_VCPU, vcpu_id) 请求创建 vCPU，内核在 kvm_vm_ioctl() 里调用 kvm_vm_ioctl_create_vcpu() 处理。

通过 create_vcpu_fd(vcpu) -> 生成新的 fd（内部也会用 anon_inode_getfile() 绑定到 kvm_vcpu_fops） -> 返回给用户态。

static const struct file_operations kvm_vcpu_fops = {
    .release        = kvm_vcpu_release,
    .unlocked_ioctl = kvm_vcpu_ioctl,
    .mmap           = kvm_vcpu_mmap,
    .llseek         = noop_llseek,
    KVM_COMPAT(kvm_vcpu_compat_ioctl),
};

3. KVM_RUN: 进入 guest / 退出（arm64）

当用户态调用 ioctl(cpu, KVM_RUN, 0) 时，kvm_vcpu_ioctl() 会将请求交给 kvm_arch_vcpu_ioctl_run() 处理（对于 ARM64）。这是 vCPU fd 上最重要的一条 control path。

3.1 ARM64 的 KVM_RUN 执行流程

kvm_arch_vcpu_ioctl_run() 的核心职责是：

  • 准备运行环境（state sync、requests 等）
  • 进入 guest（EL2/hypervisor）执行
  • 在每次 vmexit 后同步状态并填充 kvm_run
  • 直到需要返回 userspace

主循环里最关键的调用是：

ret = kvm_arm_vcpu_enter_exit(vcpu);

该调用会进入 hypervisor 路径（EL2），最终通过 kvm_call_hyp_ret(__kvm_vcpu_run, vcpu) 调用 __kvm_vcpu_run()。

__kvm_vcpu_run() 的总体职责：

  • 保存 host CPU 状态（寄存器/系统寄存器/DEBUG/FP 等）。
  • 加载 guest 状态（guest 寄存器、Stage-2 MMU、虚拟中断、timer、性能计数器等）。
  • 调用 __guest_enter()，正式“跳进” guest 执行。
  • guest 退出后恢复 host 状态并返回 exit_code 给上层（KVM 代码再处理并写入 kvm_run->exit_reason）。

退出 guest 后，kvm_arch_vcpu_ioctl_run() 会：

  • 同步硬件状态（PMU/VGIC/timer），让 kernel 和 userspace 看到一致结果。
  • 处理 exit（handle_exit() 等），把 exit 信息写入 kvm_run。

主循环会持续运行，直到 exit 需要在 host 处理（如 I/O 请求、需要重新进入 guest、signal/错误等），这时循环退出，ret 返回给调用 ioctls 的 userspace 线程。
