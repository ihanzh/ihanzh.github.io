QEMU vCPU 线程启动全流程

发表于2026-03-18|更新于2026-03-18|内核与虚拟化
|浏览量:

这篇文章从 QEMU 启动开始，沿着机器（Machine）/设备（Device）模型，一路追踪到 vCPU 线程的创建与 KVM 运行过程。

1. 从 main() 到 Machine 创建：QEMU 的大致启动流程

QEMU 采用 QOM（QEMU Object Model）构建设备树，启动时首先会创建一个 MachineState，然后在 Machine 初始化阶段创建各种设备（如 CPU、PCI 总线、内存控制器等）。

关键流程：

  1. 先通过 kvm_fd 创建 VM，获得 vmfd:

main() -> qemu_init() -> configure_accelerators() -> accel_init_machine() -> acc->init_machine() -> kvm_init() -> do_kvm_create_vm() -> kvm_ioctl(s, KVM_CREATE_VM, type)

  1. 创建 Machine:
    • main() -> qemu_init() -> qmp_x_exit_preconfig -> qemu_init_board -> machine_run_board_init()
    • machine_run_board_init() -> machine_class->init(machine)（例如 pc_init1()）
    • Machine init pc_init1() 会调用 x86_cpus_init()（其它架构也调用对应的 CPU init）
    • x86_cpus_init() 循环调用 x86_cpu_new() -> object_new(cpu_type) + qdev_realize()

✅ 结论：先创建 Machine（MachineState），然后在 Machine init 期间创建 CPU 设备（QOM Device）。

2. CPU 设备创建：从 QOM Device 到 vCPU 线程

2.1 CPU 设备的 realize 过程

CPU 是一个 QOM 设备，qdev_realize() 会触发它的 realize 回调。

CPU 设备的默认 realize 函数是 cpu_common_realizefn()（位于 cpu-common.c），它最终会调用：

  • qemu_init_vcpu(cpu)

qemu_init_vcpu() 做的事情之一是：

  • 通过 cpus_accel->create_vcpu_thread() 创建 vCPU 线程（不同加速器会有不同实现）。

所以：Machine 先创建 -> CPU 作为 Device 创建并 realize -> realize 期间启动 vCPU 线程。

2.2 KVM 加速器：kvm_start_vcpu_thread()

对于 KVM，加速器实现是 kvm_start_vcpu_thread()（kvm-accel-ops.c）。

它会调用：

accel/kvm/kvm-accel-ops.c/kvm_start_vcpu_thread()
qemu_thread_create(cpu->thread, thread_name, kvm_vcpu_thread_fn, cpu, QEMU_THREAD_JOINABLE);

这会最终调用 pthread_create():

util/qemu-thread-posix.c/qemu_thread_create()
err = pthread_create(&thread->thread, &attr, qemu_thread_start, qemu_thread_args);

2.3 QEMU 线程入口：qemu_thread_start()

qemu_thread_start() 是 QEMU 统一的线程入口包装器，它本身不做业务逻辑，负责：

  • 设定线程名（便于调试）
  • 统一 signal mask / 线程退出通知等
  • 调用真正的 start_routine(arg)
util/qemu-thread-posix.c
static void *qemu_thread_start(void *args)
{
    QemuThreadArgs *qemu_thread_args = args;
    void *(*start_routine)(void *) = qemu_thread_args->start_routine;
    void *arg = qemu_thread_args->arg;
    void *r;
    r = start_routine(arg);
    return r;
}

线程启动后进入 kvm_vcpu_thread_fn()。

3. vCPU 初始化与 KVM vCPU 创建

kvm_vcpu_thread_fn() 的第一件事是：

accel/kvm/kvm-accel-ops.c/kvm_vcpu_thread_fn()
r = kvm_init_vcpu(cpu, &error_fatal);

kvm_init_vcpu() 会调用：

accel/kvm/kvm-all.c/kvm_init_vcpu()
ret = kvm_create_vcpu(cpu);

✅ 所以：kvm_create_vcpu() 只有在 vCPU 线程已经启动后才会被调用。

kvm_init_vcpu() 的主要工作包括：

  • 通过 vm 的 KVM_CREATE_VCPU ioctl 创建一个 vCPU
  • mmap() 返回的 kvm_run 共享内存结构
  • 初始化寄存器、特权级、CPUID、MSR 等

3.1 kvm_run：QEMU ↔ KVM 的“共享缓冲区”

KVM 使用 struct kvm_run 作为用户态与内核态之间的通信区，它通过 mmap() 映射到用户态：

  1. 谁分配？

    • 内核在 KVM_CREATE_VCPU 时分配这个结构的大小与页。
    • QEMU 通过 KVM_GET_VCPU_MMAP_SIZE 得到 size，然后 mmap(..., cpu->kvm_fd, 0) 映射。

  2. 如何共享？

    • QEMU 读/写 cpu->kvm_run（用户态映射）。
    • 内核在 KVM_RUN 调用中读取输入字段（寄存器标志、IO 请求等），并在退出时写入输出字段（exit_reason、MMIO 数据、寄存器保存等）。

✅ 重要结论：这个共享区不是“并发写入的线程共享内存”。

  • 只有在 KVM_RUN ioctl 正在执行时，内核会修改 kvm_run 中的内容。
  • KVM_RUN 返回后，QEMU 读取 kvm_run 并做处理，然后准备下一次 KVM_RUN。
  • 内核不会在 QEMU 读取期间写入，除非 QEMU 再次进入 KVM_RUN。

同时 kvm_run 不是“状态同步区”，寄存器同步靠 ioctl

  • struct kvm_run 只是一个用于 KVM ↔ 用户空间通信 的共享缓冲区，主要用于传递：
    • 退出原因（exit_reason）
    • 退出相关的数据（IO/MMIO、系统事件、dirty ring、内存 fault 等）
  • 它并不包含寄存器/CPU 状态的同步数据.

✅ vCPU 状态的同步始终通过 ioctl （如 KVM_SET_* / KVM_GET_* / KVM_SET_ONE_REG 等）完成；QEMU 通过 cpu->vcpu_dirty 触发同步。

4. vCPU 主循环：进入 KVM_RUN

vCPU 线程真正执行 guest 代码的地方是主循环：

accel/kvm/kvm-accel-ops.c/kvm_vcpu_thread_fn()
do {
    qemu_process_cpu_events(cpu);           // 处理暂停/中断/热插拔等事件
    if (cpu_can_run(cpu)) {
        r = kvm_cpu_exec(cpu);             // 进入 KVM_RUN（真正执行 guest 代码）
        if (r == EXCP_DEBUG)
            cpu_handle_guest_debug(cpu);
    }
} while (!cpu->unplug || cpu_can_run(cpu));

这个循环只有在 VCPU 状态改变等特殊情况下才会退出，否则会一直运行在 kvm_cpu_exec() 中。

kvm_cpu_exec() 是 KVM 模式下让 vCPU “跑起来”的核心函数，核心逻辑可拆成几部分：

4.1 准备：拿到 kvm_run 和处理异步事件

  • struct kvm_run *run = cpu->kvm_run; 由 kvm_init_vcpu() 在创建 vCPU 时通过 mmap() 设置。
  • 检查 async 事件：kvm_arch_process_async_events(cpu)（架构相关），可能直接返回 EXCP_HLT 让 vCPU 立即回到 QEMU。

4.2 进入循环：每次 KVM_RUN 前的准备与状态同步

循环体（do { ... } while (ret == 0);）会一直执行，直到需要回到 QEMU 调度或遇到错误。

4.2.1 状态同步（dirty state）

如果 QEMU 修改了 vCPU 状态（比如通过 QMP、Debugger、MMIO 触发寄存器写等），会把 cpu->vcpu_dirty 标记为 true。

在进入 KVM_RUN 之前，QEMU 会将状态同步到内核：

accel/kvm/kvm-all.c/kvm_cpu_exec()
if (cpu->vcpu_dirty) {
    kvm_cpu_synchronize_put(cpu, KVM_PUT_RUNTIME_STATE, "at runtime");
}

kvm_cpu_synchronize_put() 底层会执行一系列 KVM ioctl（如 KVM_SET_REGS / KVM_SET_SREGS / KVM_SET_FPU / KVM_SET_ONE_REG 等）。

4.3.2 架构预处理

kvm_arch_pre_run(cpu, run);（架构相关）

4.3.3 处理“退出请求”机制（异步唤醒/停止）

如果某处需要让 vCPU 立即返回（比如暂停、热拔插、stop 等），会设置：

  • cpu->exit_request，然后调用 kvm_cpu_kick_self()，将 kvm_run->immediate_exit 置 1。

内核在下一次 KVM_RUN 时会立即返回。

4.3 进入内核：执行 KVM_RUN

最关键的一步：

accel/kvm/kvm-all.c/kvm_cpu_exec()
run_ret = kvm_vcpu_ioctl(cpu, KVM_RUN, 0);

这实际调用 ioctl(cpu->kvm_fd, KVM_RUN, NULL)，内核在此期间执行 guest，并在返回时写入 run 结构。

KVM_RUN 本质上是一个 CPU 设备的 ioctl：

  • 由 QEMU（用户态）发起，让内核在指定的 vCPU 上进入 Guest 执行。
  • 内核会在 vCPU 上“跑” guest 代码，直到发生某种退出（exit reason），然后返回用户态。

4.4 处理 KVM_RUN 返回结果（run_ret + run->exit_reason）

4.4.1 处理错误 / 信号中断（run_ret < 0）

  • -EINTR / -EAGAIN：表示被信号打断（或被 kick），QEMU 通过 kvm_eat_signals() 清理并返回 EXCP_INTERRUPT，继续调度。
  • 其他错误（除 -EFAULT 只允许用于 KVM_EXIT_MEMORY_FAULT）：通常直接报错并退出。

4.4.2 处理正常退出（run->exit_reason）

常见的 exit_reason 处理路径：

  • ✅ KVM_EXIT_IO：内核需要 QEMU 做 I/O（端口 I/O），QEMU 读 run->io 并调用 kvm_handle_io()。
  • ✅ KVM_EXIT_MMIO：Guest 访问 MMIO，QEMU 调用 address_space_rw() 进行模拟。
  • ✅ KVM_EXIT_IRQ_WINDOW_OPEN：KVM 允许 IRQ 注入，返回 EXCP_INTERRUPT 让 QEMU 继续调度（或快速重入 KVM_RUN）。
  • ✅ KVM_EXIT_SHUTDOWN：Guest 请求关机/重置，QEMU 触发 shutdown/reset，并返回 EXCP_INTERRUPT。
  • ✅ KVM_EXIT_INTERNAL_ERROR：内核内部错误，kvm_handle_internal_error() 打印信息，通常导致停止或 panic。
  • ✅ KVM_EXIT_DIRTY_RING_FULL：dirty-ring 满时，QEMU 回收 dirty-ring（kvm_dirty_ring_reap()）后继续运行。
  • ✅ KVM_EXIT_SYSTEM_EVENT：处理系统事件（如 shutdown/reset/SEV_TERM 等）。
  • ✅ KVM_EXIT_MEMORY_FAULT：内核要求 QEMU 处理内存属性转换（通常与 KVM 的私有/共享内存状态有关），然后继续。
  • ✅ 其他：交给 kvm_arch_handle_exit() 处理（如架构特定退出事件）。

4.6 结束循环：恢复状态、重新持 BQL

当 ret != 0（退出循环）时：

  • cpu_exec_end(cpu);（收尾工作）
  • bql_lock();（重新持 BQL）

如果 ret < 0，QEMU 会打印 CPU 状态并调用 vm_stop(RUN_STATE_INTERNAL_ERROR) 结束 VM。

5. 退出与清理流程

当 vCPU 退出时，会走到清理阶段：

  • kvm_destroy_vcpu(cpu)（销毁/卸载 KVM vCPU）
  • cpu_thread_signal_destroyed(cpu)
  • bql_unlock() + rcu_unregister_thread()

6. 总结：QEMU vCPU 从创建到执行的生命周期

  1. QEMU 启动 -> qemu_init() -> qemu_create_machine() -> 创建 MachineState。
  2. Machine init -> x86_cpus_init() -> object_new(cpu_type) + qdev_realize() -> 生成 CPU Device。
  3. CPU Device realize -> qemu_init_vcpu() -> cpus_accel->create_vcpu_thread() -> pthread_create() -> 进入 kvm_vcpu_thread_fn()。
  4. vCPU 线程内部：kvm_init_vcpu() -> kvm_create_vcpu() -> 启动 KVM_RUN 循环。

通过以上流程，我们可以清晰地看到：vCPU 的生命周期始于 QOM 的设备 realize，并在 vCPU 线程内部真正与 KVM 交互执行 guest 代码。
