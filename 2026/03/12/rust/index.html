无标题

发表于2026-03-12|更新于2026-03-27
|浏览量:

这句话触及了 Rust 高级编程（特别是开发底层库时）非常核心的一个概念：型变（Variance）。

这句话的通俗翻译是：

“首先，（如果我们用刚才那种简单的写法），编译器会赋予这个结构体过于严格的型变规则（Invariance，不变性）。这会导致：本来一个函数需要一个生命周期较短的 Arc<&'a str>，我们传给它一个生命周期无限长的 Arc<&'static str> 时，编译器会报错说类型不匹配。”

要听懂这句话，我们需要拆解三个概念：生命周期子类型、型变规则，以及为什么那个简单的 Arc 写法会触发这个问题。

1. 常识：长的生命周期应该能兼容短的

在正常的 Rust 代码中，生命周期较长的数据，是可以当作生命周期较短的数据来使用的。

  • 'static 意味着“活得和程序一样长”。
  • 'a 意味着“活某一段特定的时间”。
  • 显然，'static 包含了 'a。

就像你拿着 100 块钱（'static），去买 10 块钱的东西（'a），老板当然应该接受。

fn print_str(s: &'a str) { ... }

let my_str: &'static str = "hello";
print_str(my_str); // 这是完全合法的，长生命周期隐式降级为短生命周期

在正常的 Rust 中，这种“兼容性”叫做 协变（Covariance）。标准的 Box<T>、Vec<T> 和官方的 Arc<T> 都是协变的。

2. 什么是“过于严格的型变” (Strict Variance / Invariance)？

有“兼容（协变）”，就有“绝对不兼容（不变）”。
不变（Invariance） 意味着：类型必须一丝一毫都不差，哪怕你的生命周期比我长也不行。100块钱不能当10块钱用，必须正好是10块钱的纸币。

在 Rust 中，什么类型是“不变”的呢？可变引用 &mut T 和 可变裸指针 *mut T 是不变的。

为什么 *mut T 必须是不变的？（核心原因）
想象一下如果允许兼容，会发生什么灾难：

  1. 假设你有一个装着永生字符串的可变指针：ptr1: *mut &'static str。
  2. 假设 Rust 允许兼容，你把它传给了一个只要求短命字符串的指针：ptr2: *mut &'a str = ptr1;。
  3. 现在 ptr2 往这个内存里写入了一个短命的字符串。
  4. 灾难发生：ptr1 以为里面存的还是永生的字符串，当它以后去读取时，那个短命字符串可能已经被销毁了，导致内存安全漏洞（悬垂指针）。

为了防止这种“以次充好”的写入覆盖，Rust 编译器规定：只要你的结构体里包含了 *mut T 或 &mut T，整个结构体对 T 就是“不变”的，拒绝任何生命周期的兼容降级。

3. 回到你的 Arc 代码

在前文的 naive（简单的）实现中，Arc 是这么定义的：

pub struct Arc<T> {
    ptr: *mut ArcInner<T>, // <--- 罪魁祸首在这里！
}

因为你内部使用了 可变裸指针 *mut T，Rust 编译器看到了这个标志，立刻警觉起来，并给你的 Arc<T> 打上标签：“这个类型有潜在的写入风险，对 T 实施最严格的**不变（Invariance）**规则！”

4. 导致的结果（也就是那句话的意思）

因为你的 Arc 变成了“不变”的，所以当你写出这样的代码时：

// 一个普通函数，只要一个任意生命周期 'a 的 Arc
fn do_something<'a>(text: Arc<&'a str>) { ... }

fn main() {
    // 你创建了一个生命周期为 'static 的 Arc
    let my_arc: Arc<&'static str> = Arc::new("hello");

    // 报错！！！
    // 编译器：我需要 Arc<&'a str>，你给了 Arc<&'static str>。
    // 因为内部有 *mut，我拒绝把它当作较短的生命周期来处理。
    do_something(my_arc);
}

这就是那句话的意思：本来 Arc<&'static str> 代替 Arc<&'a str> 是一件非常合理且安全的事情（因为你实际上并没有提供接口去修改 Arc 里面的内容），但就因为你底层偷偷用了一个 *mut，编译器一刀切，导致库的使用者会遇到莫名其妙的生命周期类型报错，极大地影响了可用性。

5. 标准库是怎么解决这个问题的？（补充知识）

为了解决这个“编译器过于严格”的问题，Rust 提供了一个专门的类型：NonNull<T>。

NonNull<T> 底层包装了一个裸指针，但它主动告诉编译器：“请把我当成协变（Covariant）来处理！ 我向你保证我不会干那种用短命覆盖长命的坏事。”

所以真正标准库里的 Arc 是这样定义的：

use std::ptr::NonNull;

pub struct Arc<T> {
    ptr: NonNull<ArcInner<T>>, // 用 NonNull 替换了 *mut
}

这样，Arc 就恢复了正常的生命周期兼容性，Arc<&'static str> 就能顺利传给 Arc<&'a str> 的地方了。

通过我们之前的深入讨论，我们可以将 Rust 中的 裸指针（Raw Pointers，即 ptr） 的核心概念和常用方法做一个系统性的总结。

在 Rust 中，裸指针主要分为不可变裸指针 *const T 和可变裸指针 *mut T。它们是 Rust 逃生舱（Escape Hatch）的核心，专门用于绕过编译器的安全检查，进行底层的内存操作和 FFI（与 C 语言交互）。

一、 裸指针（ptr）的核心特征

与受编译器严格保护的“安全引用（&T / &mut T）”相比，裸指针具有以下特点：

  1. 不受借用检查器约束：编译器不检查它的生命周期，你可以同时拥有多个指向同一数据的可变和不可变裸指针。
  2. 不保证安全性：它可以为空（Null）、可以指向已经被释放的内存（悬垂指针）、可以未对齐。
  3. 解引用必须 unsafe：你可以随便传来传去，但只要你想读取或修改它指向的内存（如 *ptr），必须放在 unsafe {} 块中，由你个人担保安全。

二、 如何获取/创建裸指针？

  1. 从安全引用降级（强转）：
    let a = 10;
let ptr = &a as *const i32; // 安全引用强转为裸指针
  2. 使用 &raw 直接获取（跳过安全检查）：
    用于获取未对齐结构体字段或未初始化内存的地址，避免触发未定义行为（UB）。
    let ptr = &raw const data.field; // 对应旧版的 std::ptr::addr_of!()
  3. 从外部 C 函数获取：
    像 libc::malloc 这样的外部函数，返回值天生就是裸指针。
    let ptr: *mut i32 = libc::malloc(4) as *mut i32;

三、 裸指针常用的内置方法（Methods）

这些是可以直接在指针变量上调用的方法（如 ptr.xxx()）：

  1. is_null()
    • 作用：检查指针是否为空（地址是否为 0）。
    • 回顾：在你刚才 malloc 的代码里用过，用来防止内存分配失败。
  2. as_ref() / as_mut()
    • 作用：将裸指针安全地转化为 Rust 的生命周期引用（Option<&T> 或 Option<&mut T>）。如果指针是 null，则返回 None。
    • 回顾：在 Arc 的 deref 实现中，我们用 unsafe { self.ptr.as_ref() } 把裸指针变成了安全的 &ArcInner<T>。
  3. cast::<U>()
    • 作用：改变指针指向的类型（相当于 C 语言的 (U*)ptr）。比写 as *const U 更清晰。
    • 示例：let byte_ptr = ptr.cast::<u8>();
  4. offset(count) / add(count) / sub(count)
    • 作用：指针运算（Pointer Arithmetic），类似于 C 语言的 ptr + 1。注意是以元素大小为步长，而不是字节。

四、 极其重要的 std::ptr 模块级函数

这些是定义在 std::ptr 模块下的独立函数，专门用于直接操纵内存（我们在前面讲手写 Box::new 和 Drop 时重点讨论过）：

  1. ptr::read(src)
    • 作用：从内存地址中“强行拷贝”出一个值，而不触发源数据的移动或所有权转移。
    • 回顾：在手写智能指针的 drop 时，我们用它把数据临时读到栈上以便销毁（mem::drop(ptr::read(self.ptr))）。
  2. ptr::write(dst, val)
    • 作用：将一个值直接写入指定的内存地址。它会直接覆盖目标地址，而不会试图去调用目标地址原数据的 drop。
    • 回顾：在手写 Box::new 时，我们刚用 malloc 申请了一块未初始化的垃圾内存，必须用 ptr::write 填入数据。如果你用 = 直接赋值，Rust 会试图销毁那块垃圾内存，导致崩溃。
  3. ptr::copy(src, dst, count) / ptr::copy_nonoverlapping
    • 作用：按字节（bit-by-bit）拷贝内存。相当于 C 语言的 memmove 和 memcpy。
    • 注意：参数 count 是元素的个数，不是字节数。
  4. ptr::read_unaligned(src) / ptr::write_unaligned(dst, val)
    • 作用：在内存地址没有对齐（比如在 #[repr(packed)] 结构体中）的情况下安全地读写数据。普通的 *ptr 读写如果未对齐会导致直接崩溃。
  5. ptr::drop_in_place(ptr)
    • 作用：直接在指定的内存地址上就地调用该类型的析构函数（Drop），但不释放这块内存（留给 dealloc 去释放）。

五、 进阶补充：被强化过的裸指针 NonNull<T>

正如我们在 Arc 源码解析中看到的，普通的裸指针 *mut T 有两个硬伤：

  1. 可能是 Null。
  2. 会让类型失去生命周期的协变性（型变 Variance），导致生命周期报错。

因此，Rust 标准库极度推荐在开发底层数据结构时，使用 std::ptr::NonNull<T> 来替代普通的裸指针。它底层就是个裸指针，但它向编译器保证了“绝对不为空”，并且保留了对 T 的协变特性，是写高级底层库的标准配件。
